\# Contributing to Huginn SFT-Pro



Thank you for your interest in contributing to the Huginn Secure Transfer Framework. To maintain the security, modularity, and integrity of the system, please follow the protocols outlined below.



\## Security Disclosure Policy

If you identify a security vulnerability within the encryption layers or the file purging mechanism, do NOT open a public issue. Please contact the maintainer directly through the established professional channels to ensure a coordinated disclosure.



\## Technical Standards

1\. Code Quality: All submissions must follow standard Java naming conventions. Use meaningful, English identifiers for all variables, classes, and methods.

2\. Documentation: Do not use comments within the code lines unless specifically requested. All logic should be self-explanatory through clean code practices.

3\. Language: The codebase, documentation, and commit messages must be strictly in English.

4\. Modular Integrity: Ensure that any new dependency does not break the Java Module System (JPMS) configuration or create split-package conflicts.



\## Development Workflow

1\. Fork the Repository: Create a personal fork of the project to begin your work.

2\. Feature Branching: Develop your changes in a dedicated branch (e.g., feature/secure-logging).

3\. Pull Request Protocol:

&nbsp;   \* Ensure the project builds successfully using 'mvn clean install -DskipTests'.

&nbsp;   \* Provide a concise description of the changes and the problem they solve.

&nbsp;   \* Commits must be atomic and follow the conventional commit format (e.g., feat, fix, docs).



\## Author and Maintainer

This project is maintained by Arda Mecik. Any architectural changes must be approved by the maintainer to ensure alignment with the mythological and technical vision of the framework.
